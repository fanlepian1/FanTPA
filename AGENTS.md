# FanTPA Agent Rules

本文件记录FanTPA Mod的强制规范

## 1.MOD 开发规范

1. FanTPA 玩家在传送过程中不得使用任何混淆和模糊的命名，每个变量的命名不得重复
2. 所有依赖都要在mod加载的时候自动下载并且加载到classpath中
3. 所有新添加的依赖都必须保存在gradle/libs.version.toml文件中
4. 所有实现的功能必须保证格式的规范统一和代码的可读性，禁止使用过于复杂的逻辑和不必要的嵌套结构
5. 确保代码能够正常编译成功。新建的文件和类的编码格式为UTF-8

## FanTPA 模块化设计

FanTPA Mod的功能必须被划分为不同的模块，每个模块负责一个独立的功能，例如(命令模块、配置模块、语言模块、消息模块、传送模块、功能实现模块、事件模块等)

### 1. 命令模块
1. 添加命令/tpa,/tpaccept,/tpdeny,/tpahere指令，用于玩家之间的传送和确认传送请求
2. 添加指令/back，用于玩家返回死亡位置
3. 添加指令/home,/sethome,/delhome指令，用于玩家设置和管理个人传送点
4. 添加管理员指令/tpall,/tpworld指令，用于管理员进行大规模传送
5. 添加指令/fantpa主指令，用于管理员和玩家进行插件的管理和配置
6. 做好权限的分配和管理，确保不同权限的玩家只能使用相应的命令，如果有权限类插件，做好权限的兼容

### 2. 配置模块
1. 提供一个配置文件，用于玩家和管理员自定义传送相关的设置，分模块配置，配置文件要包含但不限于以下内容：
   - 传送相关的配置
   - 命令开关和参数配置
   - 权限相关的配置
   - 语言相关的配置
2. 使用异步方式加载和保存配置文件，确保不会阻塞主线程
3. 基于主指令/fantpa，可以修改配置文件中的部分参数
4. 不采用提供默认的配置文件，而是使用代码生成配置文件的方式，确保配置文件的正确性和完整性

### 3. 语言模块
1. 提供多种语言文件，用于玩家和管理员自定义插件的提示信息，例如(英文、中文、俄文等)
2. 使用占位符的方式来支持动态信息的显示，例如{player}、{replayer}、{world}、{position}等
3. 优先检测玩家使用的语言类型，如果检测不到玩家的语言类型，则使用配置中的默认语言类型对应的语言文件.
4. Mod中所有涉及到提示信息或其他消息的显示，都应该用语言文件中的内容来显示，确保插件的多样性和个性化

### 4. 消息模块
1. 提供一个消息系统，用于发送各种提示信息给玩家，例如(传送请求、传送成功、传送失败等)
2. 消息系统应该支持不同的消息类型，例如(成功、错误、警告等)，并且可以根据配置文件中的设置
3. 消息系统应该支持占位符的替换，以便在消息中动态显示相关信息，例如{player}、{replayer}、{world}、{position}等
4. 消息系统的格式可以通过语言文件进行自定义，确保消息的多样性和个性化
5. 做易于玩家能够交互的消息，例如点击消息中的玩家名字可以执行相关的命令，显示玩家坐标位置等

### 5. 传送模块
1. 提供一个传送系统，用于处理传送逻辑
2. 传送系统应该尽量支持异步执行传送操作，确保不会阻塞主线程
3. 传送系统要返回传送的结果，传入相关参数，使用多个函数来处理不同类型的传送，传入的参数可以是玩家对象、坐标、世界对象

### 6. 功能实现模块
1. 实现玩家之间的传送功能，包括请求传送、接受传送、拒绝传送等
2. 实现玩家返回死亡位置的功能
3. 实现玩家设置和管理个人传送点的功能
4. 实现管理员进行大规模传送的功能
5. 实现插件的管理和配置功能

### 7. 事件模块
1. 提供一个事件系统，用于监听和处理相关的事件，例如(玩家登录、玩家退出、玩家死亡等)
2. 事件系统应该支持注册和注销事件监听器，确保插件的灵活性和可扩展性
3. 事件系统应该提供相关的事件数据，例如玩家对象、事件类型、事件时间等，以便在事件处理器中使用
4. 事件系统应该支持异步事件处理，确保不会阻塞主线程
